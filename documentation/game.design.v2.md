# Beep.Buzz — Game Design Document v2

## Status
Pre-development. Supersedes `game.design.v1.md` and `game.plan.phase1.md` (the platformer direction). Refines `beep-buzz-v2-PRD.md`.

---

## Vision

Beep.Buzz is a 2D side-scrolling world-restoration game where Morse code is the action — not a minigame layered on top. Every cell of land the player transmits to activates and joins the world's slow transformation from grey wasteland into a solarpunk garden-city. Beep (warm, organic, generative) and Buzz (electric, technological, transformative) are gods whose currencies the player accrues and spends. Together they make the world.

---

## Core Loop

1. **Walk** to a cell.
2. **Read** its prompt (a letter, word, or phrase in Morse, displayed in a v2-style speech bubble).
3. **Transmit** it via tap/hold input on the player character.
4. **On success:** cell activates, build animation plays, currency burst animates into the HUD.
5. **Accrue** — the build generates currency over time (Beep generates, Buzz consumes/transforms).
6. **Spend** — currency unlocks new cells and higher-tier builds.
7. **Tend** — cells dim during play time if neglected; re-transmit to restore.

The morse input is the costly action. Currency is *what activated cells produce*, not what cells cost to activate.

---

## Currencies

**Beep** — warm light pulses. Generated by Beep-aligned builds (trees, gardens, solar collectors). Compounds slowly.

**Buzz** — electric charges. Generated by Buzz-aligned builds (turbines, water condensers, comms). More volatile, faster drain.

Working theory: Beep generates, Buzz consumes/transforms. Higher-tier builds need both — balanced play is required to climb tiers. Exact formulas TBD via playtest.

---

## Cells and Builds

- **Cells are hand-authored** locations along the side-scrolling map. Each has a fixed prompt that scales with player progression.
- **Cell location is fixed.** The build *at* the cell is chosen by the player from unlocked options, constrained by currency cost.
- **Prompts escalate** with player skill: single letter → short word → phrase → sentence.
- **No build placement freedom.** Cells are the world's grid; players don't paint freely.

---

## Decay

- Cells dim during *play time* only (not while offline). Fair to learners who take breaks.
- Dim cells stop generating; re-transmitting restores them.
- Creates a tending pull without punishment. Concrete tuning deferred to its own slice.

---

## Learning Model

No dedicated trainer overlay. Letters enter the player's vocabulary by playing.

- **First exposure** to a letter: cell prompt shows Morse pattern + plays audio (easiest mode).
- **After repeated correct hits:** same letter appears in later prompts as letter + audio.
- **Eventually:** letter-only (audio on demand). The player has learned it.

Existing learned-letter tracking and the DB schema from v1 carry over. The escalation rules (how many hits before a letter graduates) are TBD.

The blue/orange speech bubble look from the v2 scaffold is preserved — it's the universal prompt presentation layer.

## Multiplayer (Post-MVP)

Architecture stays compatible from day one. Built later.

- A signed-in player can visit another player's world as a guest.
- Guests can transmit unclaimed cells in the host's world.
- **In-world morse chat** — typing morse outside a cell context renders a speech bubble above the player; bubbles render across players in proximity. Same SpeechBubble styling.
- Existing `/chat` page persists for full conversations.

## Architecture

### State and persistence

- Zustand for live game state.
- **Local-first persistence:** world state in `localStorage` for unauthenticated users.
- **On sign-in:** hydrate from DB; on changes, sync via existing react-query patterns.
- **First-ever sign-in:** push local progress to DB.
- DB schema (deferred to slice 2): cells, builds, currency balances, decay timestamps, learned letters per user.

### Keep from v1

- `lib/morse.utils.ts` — encoding tables.
- The press/hold + auto-delimit timing logic from `MorseTrainer` (extracted to a shared hook).
- Speed and volume controls.
- `LearnedLetters` tracking + DB schema.
- `CustomCharacter` blob system — the player avatar.
- `Header`, `Footer`, auth flows, `/chat`, `/profile`, `/account`, `/welcome`.
- Visual design system (colors, typography, components).

### Keep from v2 scaffold (engine plumbing only — port from `v2` branch)

- `app/game/engine/useGameLoop.ts` — rAF loop.
- `app/game/world/GameWorld.tsx` — scrolling container + camera lerp pattern.
- `app/game/world/SpeechBubble.tsx` and its styling/animations — load-bearing visual asset.
- HUD overlay pattern.
- Proximity-detect + contextual interact button pattern (the cells reuse this).
- `usePlayerMovement` — adapt for click + WASD/arrows; drop gravity and jump.

### Drop from v2 scaffold

- Hearts, death, respawn (`DeathScreen.tsx`).
- Gravity, auto-jump, falling.
- Reactive/timer enemies (`Enemy.tsx`).
- Morse-locked doors (`Door.tsx`) — pattern is repurposed conceptually for cells, but the door entity itself goes.
- Trainer modal (`TrainerModal.tsx`).
- Zone 1 / The Zapper boss content.
- Hearts/HP/HUD pieces tied to combat.

## Movement

- **Click-to-walk** — primary. The world is contemplative; click commits a destination, then the player turns full attention to transmitting.
- **WASD / arrow keys** — secondary, for desktop preference and accessibility.
- No platforming, no gravity, flat ground. Camera lerps horizontally to follow the player.

## Build Order

Each slice is a shippable, demonstrable cut. No partial implementations crossing slices.

### Slice 1 — Base loop feels good

The first openspec.

- Game canvas mounted at `/`, replacing current home content. Header stays. `Learn` link renamed to `Play`.
- Side-scrolling world, flat ground, simple parallax background placeholder.
- Player = `CustomCharacter`. Movement: click-to-walk + WASD/arrows.
- HUD: Beep counter top-left, Buzz counter top-right. Both start at 0.
- Three hand-placed cells:
  - Cell 1 — prompt `E`, hardcoded build = sapling (Beep-aligned).
  - Cell 2 — prompt `T`, hardcoded build = small turbine (Buzz-aligned).
  - Cell 3 — prompt `IT`, hardcoded build = combined structure.
- Walk to a cell → speech bubble appears with prompt + interact button → transmit morse → on success: celebratory bubble animation, build animation plays, currency burst animates into the HUD, then a slow trickle starts.
- All state in Zustand. No DB. No persistence across reload.
- No decay. No NPCs. No drill. No chat-in-world. No multiplayer.

**Done when:** the loop feels good. Player can walk, transmit, see a cell come alive, watch counters tick up.

### Slice 2 — Persistence

- LocalStorage save/restore of world state on every change.
- DB schema for cells, builds, currency balances, learned letters per user.
- On sign-in: hydrate from DB. On sign-out: keep local. On first-ever sign-in: push local to DB.
- Letter learning persists across sessions.

### Slice 3 — Decay and prompt progression

- Play-time decay: cells dim past a threshold of inactive play.
- Dim cells stop generating; re-transmit restores.
- Prompt progression: as a player masters a letter, later cell prompts show it in harder modes.
- New authored cells use prompts scaled to the player's letter inventory.

### Slice 4 — Build choice at activated cells

- Activated cells offer a build menu, constrained by unlocks + currency.
- Higher-tier builds require combined Beep + Buzz cost.
- Switching builds at a cell costs a small currency penalty.

### Slice 5 — Map expansion and god presence

- More cells unlock as currency is spent.
- Hand-authored zones with visual progression: grey → muted green → vibrant.
- Beep and Buzz become visible in the sky and react to world state.

### Slice 6 — In-world morse speech bubbles

- Typing morse outside a cell context renders a speech bubble above the player.
- Bubbles use existing SpeechBubble styling and fade after a delay.

### Slice 7 — Multiplayer guests

- Signed-in players can visit another player's world.
- Guests can transmit unclaimed cells in the host's world.
- Speech bubbles render across players in proximity.
- Reuse existing chat infrastructure for matchmaking and private chat.

### Slice 8+ — Beyond MVP

- Full alphabet curriculum.
- Wardrobe / cosmetics tied to milestones.
- Authored landmarks and narrative beats.
- Sound design pass.

## Open Questions (deferred, not unilaterally resolved)

1. **Beep/Buzz mechanical asymmetry.** "Beep generates, Buzz consumes" is the working theory. Exact rates and the combined-currency cost curve need playtest tuning.
2. **Letter graduation curve.** What triggers a letter graduating from morse+audio → letter+audio → letter-only? Threshold of correct hits per session, or globally per user?
3. **Decay pacing.** How long is "neglected"? Minutes of active play? Per-cell or global?
4. **Level authoring.** Hand-authored TypeScript level files work for slice 1. A JSON format or in-app editor is deferred.
5. **God visual treatment.** Placeholder sky shapes first; animated reactive sprites later.
6. **First 60 seconds.** Onboarding for never-played-Morse users likely deserves its own slice. Placeholder for now.

## Notes for the agent

- Morse input timing and auto-delimiting are sacred. Any change requires explicit sign-off.
- Blue/orange speech bubble visuals + animations from v2 are load-bearing — preserve them.
- Persistence is local-first; no DB calls before sign-in.
- Route protection via react-query DB hooks, not middleware (per `CLAUDE.md`).
- No comments in code, all errors thrown, all errors logged with `console.error`.
